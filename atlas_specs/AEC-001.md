## PREAMBLE

Atlas is a deterministic visual rendering engine for physics education
on the Gokoo platform.

It exists for one purpose:

> Given the same inputs, Atlas must always produce the exact same image —
> on any supported machine, under the supported software stack,
> at any time, by any person.

Every architectural decision, every specification, and every line of
code must serve this purpose.

---

## PART 0 — SCOPE

### 0.1 What Atlas Is

Atlas is a Python library for generating deterministic, publication-quality
physics diagram images (PNG) for the Gokoo JEE/NEET learning platform.

Atlas governs:
- All rendering functions and geometry functions
- All style objects and design tokens
- All element specifications (block, arrow, rope, incline, etc.)
- All exported image files
- All automated tests for the above
- All specification documents in the atlas/ repository

### 0.2 What Atlas Is NOT (Non-Goals)

Atlas explicitly does not:
- Simulate physics or infer physical quantities from drawings
- Optimize diagrams by visual heuristics or AI
- Allow undocumented rendering behavior of any kind
- Serve as a general-purpose vector graphics editor
- Provide interactive or animated output in Phase 1
- Render chemistry or mathematics diagrams in Phase 1
- Make any visual decision not traceable to a frozen specification

### 0.3 Phase Scope

```
Phase 1 (current) : Matplotlib backend, PNG output only
Phase 2 (future)  : SVG output
Phase 3 (future)  : Web Canvas / interactive

No Phase 2 or Phase 3 concern may influence Phase 1 specifications
unless explicitly approved by the project owner.
```

## PART 1 — FUNDAMENTAL RULES

These rules are permanently frozen.
They cannot be changed. They cannot be overridden by any other document.
Any code or specification that violates them is invalid.

---

### Rule 1 — Determinism

**Atlas is deterministic within its supported software stack.**

Given identical inputs, Atlas must produce pixel-identical output
on any machine running the approved software stack (defined in AEC-006).

**Definition of Input (complete and exhaustive):**
Input consists of exactly these six things and nothing else:
1. Geometry parameters (numbers, angles, positions)
2. Style object (frozen dataclass)
3. Base unit value U (a single float64)
4. Export DPI (an integer)
5. Export format (string — e.g. "png")
6. Canvas size (width, height as float64 pair)

Nothing outside these six things may influence the output.

**Banned sources of non-determinism:**
- Random number generators or random seeds
- System clock or timestamps
- UUID or process ID
- Environment variables
- Unordered containers (dict iteration order, set iteration)
- Temporary filenames
- Network calls
- File system state
- OS-level font rendering differences outside the approved stack

---

### Rule 2 — Separation of Geometry and Rendering

**No rendering function computes geometry.**
**No geometry function performs rendering.**

A geometry function:
- Takes numbers and style objects as input
- Returns an immutable geometry dataclass
- Never imports or touches matplotlib or any graphics library
- Is fully testable without a display

A rendering function:
- Takes a geometry dataclass and style object as input
- Produces visual output via the approved renderer
- Never computes positions, lengths, angles, or directions
- Never modifies geometry dataclasses

---

### Rule 3 — Single Source of Truth

**Every parameter has exactly one authoritative definition.**

The authoritative source for each category:
- Dimensions     → tokens.py (via base unit U)
- Colors         → constants/colors.py (FRS-v1.0 locked palette)
- Typography     → constants/typography.py
- Style rules    → styles/ (frozen dataclasses)
- Geometry math  → geometry/ (pure functions)

**Hardcoded number policy:**
The following are exempt from this rule and may appear directly in code:
- Mathematical constants: 0, 1, -1, 0.0, 1.0, True, False
- Mathematical functions: sin, cos, tan, sqrt, pi
- Enumeration indices
- RGB normalization factor 255
- String literals that are not visual parameters

All other numbers must be named tokens. No exceptions.

---

### Rule 4 — Pixel Origin Rule

**Every visible pixel must originate from exactly one of:**

1. A Design Token (a named constant in tokens.py or constants/)
2. A Style Rule (a field in a frozen style dataclass)
3. A Geometry Rule (a mathematical formula in geometry/)
4. A Rendering Rule (a defined instruction in a rendering spec)

**For non-rendered geometry** (construction lines, guides, clip boxes):
These must originate from a Geometry Rule.
They may exist in geometry dataclasses without appearing in output.

"It looks right" is not a valid origin.
"I adjusted it by eye" is not a valid origin.

### Rule 5 — Specification Before Implementation

**No code may be written for a feature without a frozen specification.**

Strict sequence:
```
DRAFT spec
  → owner review
    → FROZEN spec
      → implementation
        → golden master test
          → done
```

A DRAFT specification may not be implemented.
Implementation may not begin until the spec status is FROZEN.

---

### Rule 6 — Completeness Before Freezing

**A specification is either COMPLETE or it does not exist.**

Complete means every field of the Element Specification Template
(AEC-002) has been filled with a specific value or formula.
Vague language, placeholders, and "TBD" entries are not permitted
in a frozen specification.

Status values: DRAFT only, or FROZEN only. Nothing in between.
Only the project owner may change status from DRAFT to FROZEN.

---

### Rule 7 — Visual Consistency Over Convenience

**Visual consistency has priority over implementation convenience.**

When a simpler implementation produces a slightly different visual
result, the correct visual result wins. Always.

**Precision vs Appearance:**
When a mathematically exact result looks visually worse than a
slightly adjusted value, the mathematically exact result wins.
Atlas does not adjust by eye. Ever.
If the exact result consistently looks wrong, the formula in the
specification must be updated, approved, and frozen — then implemented.

---

### Rule 8 — No Eye Adjustments

**No visual parameter may be set by eye or by trial and error.**

When something looks wrong, the correct sequence is:
```
1. Identify the wrong parameter by name
2. Find its token or formula in the frozen spec
3. Derive the correct value mathematically
4. Update the spec with the corrected formula
5. Get owner approval
6. Update code from spec
7. Run golden master test
8. Commit
```

Changing code directly without first updating and approving
the specification is a violation of this rule.

---

### Rule 9 — Golden Master Testing

**Every element must pass a pixel-level regression test.**

Before any element is declared implemented:
1. A golden master PNG is generated and committed to the repository
2. An automated test compares every future render against the master
3. Any difference beyond the allowed tolerance fails the test

**Golden master tolerance policy:**
```
Exact pixel match required for: positions, colors, dimensions
Tolerance of 2px allowed for: anti-aliased edges only
No tolerance allowed for: layout, proportions, label positions
```

If the golden master must change (spec was updated and approved),
the new golden master requires explicit written approval
from the project owner before being committed.

### Rule 12 — Conflict Resolution

If two frozen specifications at the same level conflict:
1. Both are immediately reverted to DRAFT status
2. The conflict is documented in writing
3. The project owner resolves the conflict
4. Both specifications are updated and re-frozen

No implementation may proceed in the conflicting area
until the conflict is resolved.

### Rule 13 — Validation Philosophy

**Atlas fails fast and loudly.**

When invalid input is detected:
- Raise the appropriate AtlasError subclass immediately
- Never silently ignore invalid input
- Never auto-correct or guess intent
- Never clip invalid geometry to valid range without raising first
- Error messages must name the invalid parameter and its value

The error hierarchy is defined in AEC-003.

### Rule 14 — Immutable Data Policy

**All specification-derived dataclasses are immutable.**

Style objects: frozen=True Python dataclasses
Geometry dataclasses: frozen=True Python dataclasses
Design tokens: module-level constants (not reassignable)

No rendering function may modify a geometry or style object.
Mutation of specification-derived data is a bug.

### Rule 15 — Dependency Direction

Dependencies flow in one direction only:

```
constants/     ← no imports from atlas
styles/        ← may import from constants/
geometry/      ← may import from constants/, styles/
elements/      ← may import from geometry/, styles/, constants/
fbd/           ← may import from elements/
core/          ← may import from elements/, styles/, constants/
```

Upward imports (e.g. geometry/ importing from elements/) are forbidden.
Circular imports are forbidden.

### Rule 16 — Canonical Ordering

Collections in geometry dataclasses use fixed canonical order:
- Corners: clockwise from bottom-left
- Forces: by zorder (lowest first)
- Layers: by zorder value ascending
- Angles: in degrees, ascending, 0° to 360°

No collection may use unordered containers (set, dict without sort)
where order affects output.

### Rule 17 — Floating Point Policy

All geometry computations use Python float (IEEE 754 float64).
No float32. No Decimal. No integer arithmetic for positions.
Trigonometric functions use math.sin, math.cos, math.tan (radians input).
All angle inputs to public functions are in degrees.
Internal conversion: radians = degrees × π / 180.
No rounding of intermediate values. Round only at final pixel output.

### Rule 18 — Performance Philosophy

**Correctness has priority over performance.**

Atlas generates static educational diagrams.
Rendering speed is not a constraint.
If a correct implementation is slower than an incorrect one,
the correct implementation is used.
No performance optimization may compromise visual correctness.

## PART 3 — DEFINITIONS

**Atlas** — The visual rendering engine for Gokoo physics education.

**Base Unit (U)** — Single float64. U = canvas_height / 10.
All dimensions are multiples of U.

**Canvas** — Drawing area from one fig_clean() call.
Fixed width × height in matplotlib units.

**Design Token** — Named constant with a unit.
Every token belongs to exactly one category (defined in S03).

**Deterministic Output** — Same inputs → pixel-identical output
on any machine in the approved software stack.

**Element** — A drawable physics object with its own spec (E01–E09).

**Frozen** — Spec status approved by project owner.
Cannot change without new version number and re-approval.

**Geometry Dataclass** — Immutable (frozen=True) Python dataclass
from a geometry function. Contains computed positions and vectors.
Contains no matplotlib objects.

**Golden Master** — Reference PNG committed to repository.
All future renders must match within the tolerance policy (Rule 9).

**Input** — Exactly: geometry params + style object + U + DPI +
export format + canvas size. Nothing else.

**Rendering Function** — Takes geometry + style, produces output.
Never computes geometry. Never modifies data.

**Specification** — Document describing one Atlas component.
Status: DRAFT or FROZEN only.

**Style Object** — Immutable (frozen=True) dataclass with visual
properties (colors, line widths, font sizes).

**Supported Software Stack** — Defined in AEC-006.
Determinism is guaranteed only within this stack.
